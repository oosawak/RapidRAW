# Decision Log

このページは、これまでの会話で固めた方針を短くまとめた記録です。

## 1. ベースにするもの

- `RapidRAW` を改造ベースにする
- 既存の UI / 構成 / 表示基盤を活かす
- 完全な新規実装ではなく、必要な部分を差し込む

## 2. GitHub Pages の役割

- `docs/index.html` を入口ページにする
- `docs/wasm-lab/index.html` を Canvas 描画ページにする
- `docs/wasm-draw/index.html` を今後の WASM 直描画ページの入口にする
- `docs/rapidraw.html` を RapidRAW 風の確認ページにする
- `localhost` は確認導線に使わない

## 3. 描画の役割分担

- 自分の描画は `WASM` で前処理する
- 他人の描画や Canvas 版の表示は `Canvas` 側で扱う
- 共有するデータは中立的な `stroke` 情報にする

## 4. WASM の担当範囲

- ストローク補間
- ブラシ色と太さの正規化
- ブラシの描画準備
- Undo / Redo の基礎
- ローカル描画の高速化

## 5. WASM に任せないもの

- 画面全体の説明 UI
- 入口ページの案内
- Canvas 表示
- 画面レイアウトの大枠

## 6. 画面の見せ方

- 描画キャンバスを主役にする
- 大きい説明カードは必要時だけ見せる
- 上部の細いステータス行は常時表示する
- 下の JSON ログは折りたたみ対象にする

## 7. 操作感の方針

- ブラシの色と太さを変更できるようにする
- 描画中のタイトルは邪魔にならない位置へ移す
- スクロールはできるだけ発生させない
- スマホでも表示エリアを確保する

## 8. 今の判断

- 現状の構成は、まだ `Canvas` 表示との分担で十分
- まずは `WASM` に描画ロジックを寄せる
- 必要になった時にだけ、さらに深く `WASM` 化する
- 今後は「全部を `WASM`」ではなく「重い処理を `WASM`」で進める

## 9. 今後の着眼点

- 筆圧や入り抜きが必要か
- Undo / Redo をどこまで `WASM` に置くか
- WebSocket 同期時のデータ形式をどうするか
- WebGPU に寄せる段階でどこを再配置するか
